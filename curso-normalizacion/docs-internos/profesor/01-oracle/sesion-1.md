1. **Crear el esquema de ejemplo** en el servidor de prácticas:

## Puntos de énfasis

### Momento clave 1: la vista como "contrato" con la aplicación

Proyectar el diagrama de la arquitectura y preguntar:

> "¿Qué pasa si mañana necesito añadir una columna `FECHA_MODIFICACION` a la tabla?"

Respuesta esperada: añado la columna a la tabla y decido si la expongo en la vista. El código .NET no se entera. Si la vista no cambia, la API no cambia, Vue no cambia.

Destacar: **la vista no es una capa de burocracia; es lo que hace que los proyectos Oracle sean mantenibles a largo plazo**.

### Momento clave 2: el ORA-00942 es el objetivo

Al ejecutar el `SELECT * FROM CURSONORMADM.TRES_TIPO_RECURSO` como CURSONORMWEB, aparece:

```
ORA-00942: table or view does not exist
```

Antes de que alguien lo vea como un error, decir claramente:

> "Esto es exactamente lo que queremos. El usuario WEB no puede ver la tabla. Si alguien roba las credenciales de la aplicación, no puede hacer DELETE directo. El error es la confirmación de que la arquitectura funciona."

### Momento clave 3: el automapeo no es magia

Mostrar la tabla de conversiones y preguntar:

> "Si la columna Oracle se llama `FECHA_PUB` y la propiedad C# se llama `FechaPublicacion`, ¿qué ocurre?"

Respuesta: el automapeo falla silenciosamente. La propiedad queda con su valor por defecto (`null` para `DateTime?`). Necesitas `[Columna("FECHA_PUB")]`.

Esto prepara el terreno para las sesiones de .NET donde se verá cómo depurar este tipo de problema.

### Momento clave 4: IDENTITY en lugar de secuencias

Mostrar la diferencia entre el patrón legacy (secuencia + trigger) y `GENERATED BY DEFAULT ON NULL AS IDENTITY`. Preguntar:

> "¿Por qué no usamos `GENERATED ALWAYS`?"

Respuesta: porque en migraciones a veces necesitas forzar un ID específico. `BY DEFAULT ON NULL` te da esa flexibilidad sin coste.

## Preguntas para provocar reflexión

Usa estas preguntas cuando el grupo esté callado o para verificar comprensión:

- *"¿Por qué el usuario WEB no se conecta directamente a las tablas?"* → Seguridad y estabilidad de contrato
- *"¿Qué convención de nombre usarías para un índice único sobre el campo CORREO de la tabla TUSUARIO?"* → `UX_TUSUARIO_CORREO`
- *"Si un campo C# se llama `NumeroLicencias` y la columna Oracle es `NUM_LICENCIAS`, ¿el automapeo funciona?"* → No. Necesita `[Columna("NUM_LICENCIAS")]`
- *"¿Cuándo usarías mapeo manual en lugar de automapeo?"* → DateOnly, BLOB, transformaciones especiales
- *"¿Qué pasa si olvidas `COMMIT` después de insertar datos de prueba?"* → Los datos existen en la sesión actual pero no son visibles para otras sesiones ni para la aplicación .NET

## Guion de la parte práctica

### Antes de la práctica (1 min)

Decir: "Vamos a conectarnos con dos usuarios distintos. CURSONORMADM es el propietario de los datos; CURSONORMWEB es como se conecta la aplicación. Quiero que veáis la diferencia en tiempo real."

### Ejercicio 1 (10 min)

1. Que los alumnos configuren las dos conexiones en su herramienta (DBeaver o SQL Developer VS Code)
2. Ejecutan el `DESCRIBE` como CURSONORMADM → ven la estructura completa
3. Ejecutan el `SELECT` sobre la vista como CURSONORMWEB → funciona
4. Ejecutan el `SELECT` sobre la tabla como CURSONORMWEB → **ORA-00942** ← parar aquí y comentarlo

### Ejercicio 2 (10 min)

1. Ejecutan la consulta de tipos (`id_tipo_recurso`, `codigo`, `nombre_es`, `nombre_ca`, `nombre_en`)
2. Pedir que identifiquen: ¿cuál de estos campos necesitaría mapeo manual para obtener `DateOnly`?
3. Pedir que expliquen por qué `CODIGO` debe ser único y qué pasa si intentan repetirlo

### Ejercicio 3 (10 min)

1. Ejecutan las queries de verificación de grants como CURSONORMWEB
2. Que confirmen que solo ven vistas y packages, no tablas directas
3. Preguntar: "¿Veis algún objeto que no debería tener grant? ¿Qué haríais?"

## Errores comunes

| Error del alumno | Causa | Cómo orientar |
|-----------------|-------|---------------|
| Configura una sola conexión con CURSONORMADM | No ha entendido la separación ADM/WEB | Explicar que CURSONORMWEB es el usuario que usa la aplicación; CURSONORMADM es el DBA |
| ORA-00942 en la vista también | No tiene el GRANT dado | Verificar que el grant está ejecutado como CURSONORMADM |
| El DESCRIBE muestra "object does not exist" | Escribe `DESCRIBE TRES_TIPO_RECURSO` sin el prefijo `CURSONORMADM.` conectado como CURSONORMWEB | Añadir el prefijo de schema |
| Busca la columna `ACTIVO` en `TRES_TIPO_RECURSO` | Arrastra el patrón genérico de entidades con estado | Recordar que esta entidad no tiene borrado lógico en el SQL real |
| No ve la salida de DBMS_OUTPUT | No ha habilitado el panel de output | En DBeaver: Enable DBMS Output en el panel de salida |

## Si sobra tiempo

- Mostrar la vista de objetos del schema en DBeaver (árbol izquierdo): tablas, vistas, packages, índices. Que los alumnos naveguen por la estructura y vean los comentarios.
- Abrir el package en DBeaver y ver la spec y el body por separado.
- Proponer: "¿Qué columna añadirías a la vista si mañana el catálogo necesitara mostrar una descripción?"

## Si falta tiempo

- Recortar el Ejercicio 3 (grants): basta con comentarlo de pasada si el tiempo aprieta.
- En el Ejercicio 1, omitir el `DESCRIBE`; ir directo al SELECT sobre la vista y el SELECT fallido sobre la tabla.
- Dejar los ejercicios 2 y 3 como trabajo individual asíncrono.

## Punto de continuidad para el profesor

Antes de cerrar la sesión, comprobar verbalmente que el grupo puede continuar con la sesión 2:

- [ ] Todos distinguen `CURSONORMADM` de `CURSONORMWEB`.
- [ ] Han visto una consulta válida sobre `VRES_TIPO_RECURSO` como WEB.
- [ ] Han visto el fallo esperado al consultar `TRES_TIPO_RECURSO` como WEB.
- [ ] Entienden que en sesión 1 no se borran ni recrean objetos.
- [ ] Entienden que la sesión 2 convierte lo observado en un script completo: tabla → vista → package → grants.

Frase de cierre sugerida:

> "Hoy no hemos construido todavía. Hemos entendido el contrato y los permisos. En la sesión 2 vamos a crear ese contrato desde cero y veremos por qué cada línea del script existe."

## Mensajes clave para cerrar

1. **ADM crea, WEB consume.** La vista es el contrato. Si la vista no cambia, la aplicación no cambia.
2. **SNAKE_CASE → PascalCase → camelCase.** `ClaseOracleBD3` lo hace automáticamente si nombramos bien.
3. **Tres conversiones especiales:** `'S'/'N' ↔ bool`, `DATE ↔ DateOnly`, `BLOB ↔ byte[]`.
4. **Grants mínimos:** `SELECT` sobre vistas, `EXECUTE` sobre packages. Nada más.
5. **En la sesión 2 lo construimos desde cero.** Esta sesión era para consolidar; la siguiente es para producir.
