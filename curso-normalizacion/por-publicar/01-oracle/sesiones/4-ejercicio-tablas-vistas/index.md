# Ejercicio sesión 2A — Diseño de vistas

::: info CONTEXTO
En la sesión 2 has visto la vista `VRES_TIPO_RECURSO` (proyección simple) y la vista `VRES_RECURSO` (con `JOIN` para evitar consultas N+1). Este ejercicio te pide diseñar **dos vistas más** del schema de reservas, decidir tú mismo cuándo aplicar `JOIN` y cuándo filtrar.

No se trata solo de escribir un `SELECT`: lo importante es la **reflexión** sobre cómo será el contrato de lectura para la aplicación .NET que construirás en las próximas sesiones.
:::

[[toc]]

## Por qué este ejercicio

Las vistas son sencillas técnicamente, pero el diseño no es obvio: cada decisión (alias, `JOIN`, filtro de `ACTIVO`, `NVL`) impacta en el código .NET que las consume. Aquí trabajas el **criterio**, no la sintaxis.

Cuando termines, compararás tu propuesta con las vistas reales del schema (`SQL/CURSONORMADM/VISTAS/VRES_HORARIO_DIA.sql` y `VRES_FRANJA_HORARIO.sql`) para discutir las diferencias.

## Objetivos

Al terminar el ejercicio debes ser capaz de:

- Decidir qué columnas expone una vista a la aplicación y con qué alias.
- Justificar cuándo añadir un `JOIN` con otra tabla y cuándo dejarlo en la aplicación.
- Decidir si la vista filtra `WHERE ACTIVO = 'S'` o expone el flag.
- Documentar una vista siguiendo el formato visto en el schema real (cabecera `/* ... */` con propósito, IMPORTANTEs y bitácora).
- Conceder `GRANT SELECT` al usuario WEB.

## Tablas de partida

Trabajas sobre dos tablas reales del schema:

::: code-group

```sql [TRES_FRANJA_HORARIO]
CREATE TABLE CURSONORMADM.TRES_FRANJA_HORARIO (
    ID_FRANJA    NUMBER       GENERATED BY DEFAULT ON NULL AS IDENTITY,
    ID_RECURSO   NUMBER       NOT NULL,
    FECHA_INICIO DATE         NOT NULL,
    FECHA_FIN    DATE         NOT NULL,
    ACTIVO       VARCHAR2(1)  DEFAULT 'S' NOT NULL,
    --
    CONSTRAINT PK_TFRANJA_HORARIO       PRIMARY KEY (ID_FRANJA),
    CONSTRAINT FK_TRES_FRANJA_RECURSO   FOREIGN KEY (ID_RECURSO)
        REFERENCES CURSONORMADM.TRES_RECURSO (ID_RECURSO),
    CONSTRAINT CK_TRES_FRANJA_HORARIO_ACTIVO CHECK (ACTIVO IN ('S', 'N')),
    CONSTRAINT CK_TFRANJA_HORARIO_FECHAS     CHECK (FECHA_FIN >= FECHA_INICIO)
);
```

```sql [TRES_HORARIO_DIA]
CREATE TABLE CURSONORMADM.TRES_HORARIO_DIA (
    ID_HORARIO_DIA NUMBER       GENERATED BY DEFAULT ON NULL AS IDENTITY,
    ID_FRANJA      NUMBER,            -- opcional: NULL = horario genérico de recurso
    ID_RECURSO     NUMBER,            -- usado en horarios genéricos
    DIA            NUMBER,            -- 1..7, opcional
    HORA_INICIO    NUMBER       NOT NULL,
    MINUTO_INICIO  NUMBER       NOT NULL,
    HORA_FIN       NUMBER       NOT NULL,
    MINUTO_FIN     NUMBER       NOT NULL,
    ORDEN          NUMBER,
    BLOQUEADO      VARCHAR2(1)  DEFAULT 'S' NOT NULL,
    --
    CONSTRAINT PK_TRES_HORARIO_DIA            PRIMARY KEY (ID_HORARIO_DIA),
    CONSTRAINT FK_TRES_HOR_DIA_FRANJA         FOREIGN KEY (ID_FRANJA)
        REFERENCES CURSONORMADM.TRES_FRANJA_HORARIO (ID_FRANJA),
    CONSTRAINT CK_TRES_HORARIO_DIA_BLOQUEADO  CHECK (BLOQUEADO IN ('S', 'N')),
    CONSTRAINT CK_TRES_HORARIO_DIA_DIA        CHECK (DIA IS NULL OR DIA BETWEEN 1 AND 7),
    CONSTRAINT CK_TRES_HORARIO_DIA_HORA_INICIO CHECK (HORA_INICIO BETWEEN 0 AND 23),
    CONSTRAINT CK_TRES_HORARIO_DIA_HORA_FIN    CHECK (HORA_FIN    BETWEEN 0 AND 23),
    CONSTRAINT CK_TRES_HORARIO_DIA_MIN_INICIO  CHECK (MINUTO_INICIO BETWEEN 0 AND 59),
    CONSTRAINT CK_TRES_HORARIO_DIA_MIN_FIN     CHECK (MINUTO_FIN    BETWEEN 0 AND 59)
);
```

:::

::: warning IMPORTANTE
Una franja **siempre** pertenece a un recurso (`ID_RECURSO NOT NULL`). Un horario por día puede pertenecer:

- a una **franja** (`ID_FRANJA` informado): horario específico de un periodo.
- a un **recurso genérico** (`ID_FRANJA NULL` + `ID_RECURSO` informado): horario por defecto del recurso.
:::

## Ejercicio 1 — `VRES_FRANJA_HORARIO`

Diseña una vista que permita a la aplicación .NET consultar las franjas horarias asociadas a los recursos.

### Mínimo que debe exponer

- Identificador de la franja.
- Identificador del recurso.
- Fecha de inicio.
- Fecha de fin.
- Indicador de si la franja está activa.

### Decisiones a tomar

- ¿Conviene un `JOIN` con `TRES_RECURSO` para devolver `NOMBRE_RECURSO_ES` / `NOMBRE_RECURSO_CA` / `NOMBRE_RECURSO_EN`? Justifica.
- ¿Filtrarías `WHERE ACTIVO = 'S'` en la vista o dejarías que la aplicación decida con `WHERE ACTIVO = :activo`?
- ¿Aplicarías `NVL(ACTIVO, 'N')` para no devolver `NULL` aunque el `NOT NULL` ya lo garantice?
- ¿Hace falta un alias diferente al nombre de columna (por ejemplo `FECHA_INI` en lugar de `FECHA_INICIO`) o el automapeo .NET ya funciona con el nombre tal cual? *(Pista: revisa el ejemplo del `ClaseFranjaHorario` C# del proyecto.)*

### Formato de cabecera (obligatorio)

Documenta la vista con el formato del schema real:

```sql
SELECT /* Vista de <propósito>.

         IMPORTANTE: <invariante o decisión clave>.

  YYYY-MM-DD  AUTOR  Descripción del cambio.
                       */
  COLUMNA1,
  COLUMNA2,
  ...
FROM CURSONORMADM.TRES_TABLA;
```

### Entregable parcial

```sql
CREATE OR REPLACE FORCE VIEW CURSONORMADM.VRES_FRANJA_HORARIO (...) AS
  SELECT /* ... */
    ...
   FROM CURSONORMADM.TRES_FRANJA_HORARIO ...;

GRANT SELECT ON CURSONORMADM.VRES_FRANJA_HORARIO TO CURSONORMWEB;
```

## Ejercicio 2 — `VRES_HORARIO_DIA`

Diseña una vista que permita consultar los tramos horarios configurados para cada día, tanto de franjas como genéricos por recurso.

### Mínimo que debe exponer

- Identificador del horario del día.
- Identificador de la franja horaria (puede ser `NULL` si es genérico).
- Identificador del recurso (informado al menos en horarios genéricos).
- Día de la semana.
- Hora y minuto de inicio.
- Hora y minuto de fin.
- Indicador de si el tramo está bloqueado.
- Orden de visualización.

### Decisiones a tomar

- ¿Aplicarías `NVL(BLOQUEADO, 'N')` igual que `VRES_RECURSO` hace con `ACTIVO`? Mira el código C# (`SelectHorarioDiaColumns` en `HorarioDiaService.cs`) que usa exactamente `NVL(BLOQUEADO, 'N')`. ¿Pondrías ese `NVL` en la vista o seguirías dejándolo en el SELECT del servicio?
- ¿Conviene un `JOIN` con `TRES_FRANJA_HORARIO` para incluir `FECHA_INICIO_FRANJA` y `FECHA_FIN_FRANJA`? Razona el coste: la mayoría de horarios son genéricos (`ID_FRANJA NULL`), así que el `JOIN` sería `LEFT JOIN`.
- ¿Conviene un `JOIN` con `TRES_RECURSO` para incluir el nombre del recurso? El `ID_RECURSO` puede ser `NULL` cuando el horario va por franja: el `JOIN` también sería `LEFT JOIN`.
- ¿Ordenarías la vista por `DIA, ORDEN` o lo dejas a la aplicación?

::: tip BUENA PRÁCTICA
Si la mayoría de consumidores van a hacer `WHERE ID_FRANJA IS NULL AND ID_RECURSO = :id ORDER BY DIA, ORDEN`, conviene **no** ordenar en la vista (cada cliente añade su propio `ORDER BY`). En cambio, los `NVL` son baratos: se evalúan por fila y simplifican el código de cada servicio que la consume.
:::

### Punto de reflexión obligatorio

Responde por escrito:

1. ¿La vista debería mostrar solo identificadores o también datos descriptivos del recurso?
2. ¿Tiene sentido hacer `JOIN` con la tabla de recursos sabiendo que a veces `ID_RECURSO` es `NULL`?
3. ¿Filtrarías únicamente franjas activas o dejarías que la aplicación decida?
4. ¿Cómo documentarías la finalidad de cada vista para que un compañero pueda mantenerla sin hablar contigo?

## Comparación con el schema real

```sql
  FROM all_views
 WHERE owner = 'CURSONORMADM'
   AND view_name IN ('VRES_FRANJA_HORARIO', 'VRES_HORARIO_DIA')
 ORDER BY view_name;
```

:::
